## 摘要（实验前版本）

对地观测卫星调度需要在随状态动态变化的“任务—卫星—观测窗口”候选集中进行长序列决策；一个局部动作会通过姿态转移、窗口冲突与星上资源占用改变后续可行域。异构求解器的最终排程可提供高质量示范，但将其规范重放为构造轨迹时只暴露被执行动作，普通行为克隆因而无法区分未选可行动作之间不同的下游机会代价。本文提出冲突覆盖机会代价蒸馏方法 CCOD。首先，将异构求解器输出转换为可哈希、可精确重放的 solution-induced constructive traces，并固定目标函数、候选上限与稳定动作顺序。其次，在每个状态仅查询固定预算的候选动作，采用确定性的有界视域 continuation 估计短期回报；同时利用未来窗口删除、优先级质量损失、最后可行机会、姿态转移余量与资源压力构造 EOS conflict/resource footprint，据此分配反事实查询预算。最后，使用已查询动作对的相对回报训练集合条件的单头评分器；部署时对完整 capped feasible set 一次打分，不执行 rollout、树搜索或在线求解器调用。实验将基于 EOS-Bench 按实例划分数据，在相同查询预算与 continuation 下比较 uniform、policy、top-k 和通用多样性选择器，并联合报告双尾覆盖、排序质量、完整排程目标、可行率、在线时延与离线标注成本。正式实验尚未完成，因此本摘要不预报效果数值；方法结论受预先定义的 100-state 与 500-state Go/No-Go 门槛约束。

**关键词：** 对地观测卫星调度；反事实监督；机会代价；模仿学习；查询效率；排序蒸馏

## Abstract (proposal version; results pending)

Earth-observation satellite scheduling requires decisions over large, dynamic, and strongly constrained candidate sets while balancing profit, completion, timeliness, and resource utilization. Solver-induced demonstrations reveal only selected actions and therefore do not expose the downstream opportunity costs of feasible alternatives. We propose **Conflict-Covered Opportunity-Cost Distillation (CCOD)**, a resource-aware offline learning framework that reconstructs exact scheduling prefixes, allocates a fixed counterfactual query budget using EOS conflict and resource footprints, estimates action values with deterministic finite-horizon continuations under a unified objective, and distills the resulting pairwise order into a lightweight set-conditioned policy with search-free deployment. The planned evaluation uses instance-level splits on EOS-Bench and jointly measures label validity, query efficiency, complete-schedule quality, feasibility, online runtime, and Mac-CPU labeling cost. No empirical claim is inserted before the preregistered diagnostic and held-out evaluation gates are passed.

## 主要创新与声明边界

以下三项是待实验验证的论文主张；只有对应 gate 通过后，才改为确定语气。

1. **面向 selected-only solver schedules 的可复现监督协议。** 将异构求解器最终排程明确界定为 canonical solution-induced constructive traces，而非求解器真实内部轨迹；通过精确前缀重放、稳定动作键、capped dynamic action set、统一目标和 instance-level split，把终局排程转化为可复核的 partial cost-vector supervision。该项是 EOS 场景的数据语义与实验协议，不声称 exact replay 本身是通用算法创新。
2. **约束一致的 EOS 结构感知查询分配。** 提出 conflict/resource footprint，用候选动作导致的未来窗口删除、优先级质量损失、最后机会损失、姿态与窗口余量、星上资源瓶颈刻画其潜在机会代价；在固定预算下联合覆盖高风险端、低风险端、结构多样性及少量 seeded-uniform 支持。核心主张不是 farthest-first 或主动采样本身，而是 EOS 约束一致的 footprint 能否提高单位反事实查询的双尾覆盖和排序监督效率。
3. **带完整成本核算与证伪机制的离线机会代价蒸馏闭环。** 在精确状态上以固定预算生成 bounded-continuation 相对排序标签，训练集合条件的单头评分器，并在完整 capped feasible set 上实现无搜索推理；同时用 exhaustive small states、strong-continuation rank check、selector-only footprint permutation 和预算配对实验，把“标签质量—离线计算—完整排程收益—在线时延”置于同一评价闭环。这里的贡献是 EOS selected-only 数据条件下的组合与验证，不声称 cost-to-go、pairwise ranking 或 search-free distillation 是新的。

新颖性核验的保守结论为 **Level 2 — High Overlap**：SeaRNN、LOLS、AggreVaTe/AggreVaTeD 已覆盖通用 cost-to-go/rollout supervision，Herrmann–Schaub 已覆盖 EOS 中从 MCTS 动作价值到无搜索策略的蒸馏。因此论文的成立点必须是上面三项的精确组合及其可测收益，而不是泛称“反事实学习”或“regret distillation”。

## 0. 一页结论

这篇论文不再声称“首次使用替代动作 rollout、cost-to-go、regret 或无搜索蒸馏”。SeaRNN、LOLS、AggreVaTe/AggreVaTeD 和 EOS 调度中的 MCTS-Q 蒸馏已经覆盖了这些一般机制 [3, 7–10]。

论文唯一需要负重的方法性主张是：

> 在 EOS-Bench 的动态大候选动作集中，候选动作具有可由冲突图、窗口稀缺度、姿态转移余量和星上资源瓶颈描述的结构。用该结构分配固定的反事实查询预算，能比 uniform、policy sampling 和 top-k 查询更高效地覆盖机会代价的两端，并把这种排序知识蒸馏为部署时不做搜索的轻量策略。

最小方法闭环是：

1. 将异构 solver 的最终 schedule 规范化为可确定性重放的 **solution-induced constructive traces**；
2. 在精确前缀上，对每个状态只查询至多 $B=16$ 个动作；
3. 用 EOS conflict/resource footprint 选择这些动作；
4. 用统一目标下的 bounded continuation 得到 signed advantage；
5. 用单头、集合条件的 scorer 学习 queried pairs 的排序；
6. 部署时对枚举出的全部合法动作一次前向打分并取 `argmax`，不做 rollout、MCTS 或在线 solver 调用。

当前最重要的工作不是租 GPU，而是在已经通过的 exact replay、约束与目标基础上，实现 `force_action`、$H=5$ continuation、数据划分和可辨识表示。现有 BC 只有约 19K 参数，Mac CPU 每个 epoch 约 38–40 秒，尚不存在 GPU 瓶颈。

## 1. 论文定位、边界与目标读者

### 1.1 推荐定位

目标是航天调度、运筹优化、智能优化或应用型 AI 论文，而不是把它包装成通用序列学习新理论。

建议叙事顺序：

1. EOS 动作空间大、动态变化且强约束；
2. solution-induced trace 只告诉模型“做了什么”，没有告诉模型“其他动作会损失什么”；
3. 对所有动作做 continuation 太贵；
4. EOS 的冲突/资源结构可以帮助选择最值得查询的动作；
5. 结构化有限查询能以较低离线代价改善完整 schedule，同时保持在线推理轻量。

### 1.2 核心研究问题

> 如何从 heterogeneous solver final schedules 构造的规范轨迹中，以极少的 EOS 结构感知查询恢复未执行动作的机会代价排序，并将其蒸馏为无需在线搜索的卫星调度策略？

### 1.3 明确不声称的内容

- 不声称 forced alternative action rollout 是新的；
- 不声称 local regret / cost-to-go imitation 是新的；
- 不声称 random、policy sampling、top-k、uncertainty 或 generic diversity sampling 是新的；
- 不把双头网络、softplus regret、$z-\beta r$ 或某个损失组合列为核心贡献；
- 不把 MIP、SA、GA、ACO 的 final schedule 误称为其内部逐步决策轨迹；
- 不在主方法中加入 learner-state aggregation；它只在主结果已经为正、且 off-trace shift 被证实后作为后续扩展；
- 不承诺跑完 EOS-Bench 全部实例，也不把 JSONL 行数当作独立样本数。

### 1.4 相关工作与创新边界

#### EOS 调度与学习型求解

敏捷 EOS 排程的经典问题定义与二十年方法综述分别见 Lemaître 等 [12] 和 Wang 等 [11]。EOS-Bench 统一了多种敏捷/非敏捷场景、约束、求解器和五维评测指标，是本文的任务与复现基座 [1]。既有学习方法已经覆盖多个相邻方向：Hadj-Salah 等用深度强化学习处理简化的观测获取调度 [2]；Herrmann–Schaub 用 MCTS 生成状态–动作价值并回归为可在星上快速执行的无搜索价值网络 [3]，并进一步研究敏捷 EOS 的强化学习调度 [13]；Jacquet 等用 GNN/PPO 表示大规模候选冲突，并在部署阶段继续使用 MCTS 改善解 [4]；近期工作还研究了跨规模泛化的资源约束 DRL [5]。因此本文不能声称“首次把学习用于 EOS 调度”“首次蒸馏搜索价值”或“首次处理动态候选集”。

#### Cost-to-go imitation 与 learning to search

DAgger 说明了行为克隆在 learner-induced state distribution 上的复合误差问题 [6]；AggreVaTe 进一步用专家 cost-to-go 区分不同动作错误的后果 [7]。LOLS 对合法替代动作执行 rollout，并以 rollout loss 相对最佳动作的差定义局部代价 [8]；Deeply AggreVaTeD 将训练期动作价值用于可微神经策略 [9]；SeaRNN 则在大词表结构预测中采用 forced alternatives、完成式 rollout、局部 cost vector 和 uniform/policy/top-k 子采样，同时保持测试时不做 rollout [10]。这些工作已覆盖 CCOD 的通用学习骨架。

#### 本文相对差异

本文的可辩护 delta 不是单个通用算子，而是：从只暴露最终 assignment 集合的异构 EOS solver 解中构造可验证状态；在超大、动态、受真实约束过滤的 task–satellite–window 候选集上，用 EOS conflict/resource footprint 分配固定反事实预算；以标签成本、双尾覆盖和完整 schedule 收益共同检验该分配是否优于 SeaRNN-style 通用采样。最接近的单篇方法威胁是 SeaRNN，最接近的局部 regret 目标是 LOLS，最接近的同域价值蒸馏工作是 Herrmann–Schaub。只有这个组合在公平预算下带来显著收益，CCOD 的方法贡献才成立。

### 2.2 三种必须区分的数据对象

1. **Solver final schedule**：求解器最后给出的 assignment 集合；
2. **Canonical solution-induced constructive trace**：按固定任务顺序重新播放 final schedule 得到的构造轨迹；
3. **Faithful solver trajectory**：求解器真实执行时的内部逐步决策。

当前 MIP priority replay、SA、GA、ACO 数据属于第 2 类，不属于第 3 类。全文统一使用 “solution-induced constructive trace”。

### 2.3 独立样本单位

独立样本单位是 scenario/instance，而不是 JSONL row，也不是 solver schedule。

- 同一个 instance 的全部 solver 解、全部目标设置和全部 trace row 必须进入同一 fold；
- 先按 instance 划分 train/dev/test，再抽取状态；
- bootstrap 和显著性检验以 instance 为 cluster；
- 对相同 assignment prefix 做 canonical hash 去重，避免同一状态被多个 solver 反复计数；
- final test 在方案冻结后才打开，100-state diagnostic 只使用 dev instances。

当前 11 个本地场景足够做方法诊断，不足以支持强跨规模结论。若 500-state gate 通过，再从 EOS-Bench 选择或生成一个紧凑的多规模集合；目标为 3 个规模档、共 30–60 个独立 instances，而不是追求全量 benchmark。

## 3. 可证伪假设

### H1：机会代价异质性

在相同精确状态下，不同合法动作的 $Q_H(s,a)$ 存在有实际意义的差异；否则 one-hot BC 已足够，本文动机不成立。

### H2：结构可预测性

候选对应卫星的资源状态、窗口稀缺度、姿态转移余量和未来冲突 footprint 能预测动作间 $Q_H$ 排序；否则昂贵标签无法被轻量模型泛化。

### H3：查询效率

在相同 $B$、相同 continuation、相同模型、相同训练步数下，CCOD 的 footprint selector 比最佳 uniform/policy/top-k/generic-diversity selector 覆盖更多真实高代价与低代价动作。

### H4：完整调度收益

CCOD 学到的单头 scorer 在 held-out instances 上提高完整 schedule 的 $F_\omega$，同时 feasibility 不下降、在线推理仍接近 BC。

### 3.1 唯一负重变量与负对照

对状态 $s$，令查询集合为 $U_B(s)$；令 $q_s=\max\{1,\lceil0.1|\mathcal A_C(s)|\rceil\}$，并按 stable candidate key 处理 $Q_H$ 并列。记真实得分最高和最低的 $q_s$ 个动作为 $\operatorname{Best}_{q_s}(s)$ 与 $\operatorname{Worst}_{q_s}(s)$。主证伪变量定义为 exhaustive dev states 上的：

$$
\begin{aligned}
\operatorname{TailRecall}@B(s)
&=\frac{1}{2}\left(
\frac{|U_B(s)\cap\operatorname{Best}_{q_s}(s)|}{q_s}
+
\frac{|U_B(s)\cap\operatorname{Worst}_{q_s}(s)|}{q_s}
\right),\\
V&=\frac{1}{|\mathcal S_{\mathrm{dev}}|}
\sum_{s\in\mathcal S_{\mathrm{dev}}}\operatorname{TailRecall}@16(s).
\end{aligned}
$$

只看最差动作会忽略“是否查询到好动作”，因此使用双尾覆盖；候选不足 10 个时由 $q_s$ 的下界保证两端至少各取一个动作。

唯一主负对照是：

> 在每个状态内随机置换 $v_s(a)$ 与 candidate key 的映射，且只在 selector 内置换；模型仍接收真实特征，$B$、随机种子、真实 continuation 标签、网络和优化步骤全部不变。

若置换后 $V$、NDCG 和每 1,000 次查询带来的完整 schedule 增益没有回落到 uniform 区间，则“EOS footprint 改善查询效率”的解释被证伪。

## 4. 问题定义

### 4.1 状态、动作与 cap

在状态 $s_t$，数学上的全部合法动作集合记为：

$$
\mathcal A(s_t).
$$

实际候选生成器受 `max_candidates` 上限 $C$、时间离散和枚举顺序影响，论文真正使用的集合记为：

$$
\mathcal A_C(s_t)\subseteq \mathcal A(s_t).
$$

`SKIP` 是普通合法动作。论文所有“完整动作集”表述均指完整的 $\mathcal A_C(s)$，不得把 cap 后集合写成理论上的 $\mathcal A(s)$。

要求：

- cap 必须记录在每个 run manifest；
- candidate key 与 tie-break 顺序稳定；
- trace action 不在 $\mathcal A_C(s)$ 时不得在 validation/test 中偷偷注入；
- 小实例上比较多个 $C$，确认 cap 不改变主要参考排序后再冻结 $C$。

### 4.2 规范构造轨迹

给定 final schedule $Y$，固定：

1. task priority 降序；
2. priority 相同时按稳定 task id；
3. 若 $Y$ 为当前 task 分配 assignment，则重放该 candidate key；
4. 否则执行 `SKIP`。

由此得到：

$$
\tau(Y)=\{(s_t,a_t^{\mathrm{obs}})\}_{t=1}^{T}.
$$

$a_t^{\mathrm{obs}}$ 表示 trace observed action，不写成求解器真实在线动作。

### 4.3 统一目标

最终目标使用 EOS-Bench 的归一化多指标组合：

$$
F_\omega(S)
=
\omega_p\,\mathrm{Profit}(S)
+\omega_c\,\mathrm{Completion}(S)
+\omega_t\,\mathrm{Timeliness}(S)
+\omega_b\,\mathrm{Balance}(S),
$$

其中：

$$
\omega_p+\omega_c+\omega_t+\omega_b=1,
\qquad
\omega_i\ge 0\quad(i\in\{p,c,t,b\}).
$$

所有 trace 质量、counterfactual label 和最终评估必须调用同一 `ObjectiveModel` 和同一 $\omega$。不再给 `SKIP` 额外添加 $-1$，也不再把 $10\times\Delta F$ 与程序性惩罚混合。

主实验先冻结 balanced objective；profit-first 和 completion-first 只作为较小的目标条件泛化实验。模型输入显式包含 $\omega$。

- `future_return_H` 改名为 `observed_return_H`，仅表示 behavior trace 中观测动作的 $H$ 步回报，不作为 counterfactual label；
- counterfactual label 只由统一 $F_\omega$ 计算；
- 同一 final schedule 的逐步增量之和与最终目标变化做回归测试。

### P0.4 动作集语义

- 记录 cap 前候选数、cap 后候选数、feasibility 后候选数；
- 明确当前 cap 发生在 feasibility filtering 前的影响；
- validation/test 使用完整 $\mathcal A_C(s)$，不使用“抽 256 个并强制加入 observed action”的集合；
- 小状态上检查 `C` 的排序稳定性。

### P0.5 表示可辨识性

当前 5 维 state + 10 维 candidate 的独立 MLP 会把不同卫星但标量相同的候选映射成同一分数。昂贵标签之前必须补充：

- candidate 所属 satellite 的分组信息，而不是依赖任意全局 ID；
- sensor/platform type embedding；
- 当前卫星剩余 storage、power、已排任务数、最近 assignment、姿态与时间；
- predecessor/successor transition slack；
- window slack、该任务剩余机会数；
- future conflict count、priority mass、last-window loss；
- objective weights。

### P0.6 划分与去重

- 按 instance group split；
- 同一 instance 的所有 solver/objective 数据同 fold；
- canonical prefix hash 去重；
- quality-weighted BC 只在相同 `(instance, objective)` 组内归一化；
- final test manifest 冻结并封存。

## 6. 方法：CCOD

### 6.1 精确 counterfactual continuation

对状态 $s$ 和被查询动作 $a\in\mathcal A_C(s)$：

1. deep-copy 精确 partial schedule；
2. 强制执行 `a`；`SKIP` 保持当前任务未分配；
3. 使用固定的 deterministic objective-greedy policy 继续到总共 $H$ 个决策；强制动作计作第 1 步，随后最多执行 $H-1$ 个 greedy 决策；
4. 所有 ties 用 stable candidate key 打破。

主配置：

$$
H=5.
$$

令 $S(s)$ 表示状态 $s$ 对应的 partial schedule，令 $\mathcal C_H(s\oplus a)$ 表示先强制执行 $a$、再按上述策略完成总计至多 $H$ 个决策后得到的 schedule。若剩余任务少于 $H$，则在任务耗尽时提前终止。定义：

$$
Q_H(s,a)
=
F_\omega\!\left(\mathcal C_H(s\oplus a)\right)-F_\omega\!\left(S(s)\right),
$$

以及相对 observed action 的 signed advantage：

$$
A_H(s,a)
=
Q_H(s,a)-Q_H(s,a^{\mathrm{obs}}).
$$

这里没有 budget-set best reference、独立 `SKIP` penalty 或 selector-dependent normalization。同一状态的 pairwise 差值为：

$$
A_H(s,a)-A_H(s,b)=Q_H(s,a)-Q_H(s,b).
$$

只在完整枚举的小型诊断状态上，定义用于评估的非负机会代价：

$$
\operatorname{OC}_H(s,a)
=
\left[\max_{b\in\mathcal A_C(s)}Q_H(s,b)\right]-Q_H(s,a).
$$

$\operatorname{OC}_H$ 表示在已冻结 capped action set $\mathcal A_C(s)$ 上定义的 opportunity cost；它会随候选上限 $C$ 改变，因此 cap 必须进入 run manifest。该量只用于定义 high-cost/worst-decile coverage，不作为规模化训练时依赖“预算集合内最大值”的标签。

在小型 exhaustive dev 子集上，再用 beam-8 或可解的 MIP continuation 检查 cheap continuation 的排序稳定性。主方法不在线调用它们。

### 6.2 EOS conflict/resource footprint

对候选动作 $a$，构造无 counterfactual rollout 的结构签名：

$$
v_s(a)=
\left[
\begin{array}{c}
\text{immediate normalized gain}\\
\text{deleted future-window count}\\
\text{deleted future priority mass}\\
\text{last-feasible-window loss count}\\
\text{remaining storage / power ratios}\\
\text{storage / power consumption ratios}\\
\text{transition slack}\\
\text{window slack}\\
\text{same-satellite contention}\\
\text{opportunity scarcity}
\end{array}
\right].
$$

令 $\mathcal O_L(s)$ 为规范任务顺序中后续至多 $L$ 个任务在状态 $s$ 下的可行 observation opportunities。未来冲突集合定义为：

$$
\mathcal D_L(s,a)
=
\left\{o\in\mathcal O_L(s):
o\text{ 在 }s\text{ 下可行，且在 }s\oplus a\text{ 下不可行}\right\},
$$

其中只扫描后续固定 $L$ 个任务的 opportunities，删除关系由与 rollout 相同的 constraint checker 判断。$L$ 在 dev profiler 后冻结，默认从 32 开始。该计算可用时间区间与 satellite 索引缓存，不能退化成全局 $O(|\mathcal A|^2)$ 暴力比较。

签名各维只用 train split 统计量做 robust scaling；不根据 counterfactual label 调 selector 权重。

### 6.3 Conflict-covered query allocation

selector 输出一个稳定、可取前缀的候选排序。主预算：

$$
B=16.
$$

排序构造规则：

1. 必含 $a^{\mathrm{obs}}$；
2. 必含合法 `SKIP`；
3. 加入结构风险最大与最小候选；
4. 在预先固定的位置加入 4 个 seeded-uniform 候选，用于保持支持和测量选择偏差；
5. 其余位置在 robust-scaled $v_s(a)$ 空间做 deterministic farthest-first；
6. 去重后不足 $B$ 时按 stable-key uniform ranking 补齐。

结构风险只由 conflict mass、last-window loss、资源瓶颈与 slack 的固定无量纲组合得到。查询 $B\in\{4,8,16\}$ 时取同一排序前缀，使预算消融可复用标签。

该 selector 的贡献不是 farthest-first 算法本身，而是 EOS constraint-consistent footprint、风险两端覆盖和固定预算下的可证伪 query-efficiency。

### 6.4 单头集合条件 scorer

所有学习方法共享同一个轻量模型：

1. candidate encoder 编码 task、window、platform/sensor、资源和 footprint；
2. per-satellite encoder 聚合当前卫星的 assignments 与剩余资源；
3. DeepSets 用 mean + max pooling 得到候选集合上下文；
4. 单头 scorer 输出：

$$
f_\theta(s,a)\in\mathbb R.
$$

模型参数目标小于 0.5M。不同卫星候选通过其 per-satellite context 区分，不依赖任意 satellite id 的记忆。

### 6.5 排序蒸馏

先用 BC 初始化 scorer。随后只使用 queried actions 的所有非平局 pairs：

令 $\mathcal P_s$ 为查询集合中所有满足 $A_H(s,a)\ne A_H(s,b)$ 的无序动作对；每对按 stable candidate key 固定方向。若 $|\mathcal P_s|=0$，则该状态不计入排序损失。

$$
y_{ab}
=
\operatorname{sign}\!\left(A_H(s,a)-A_H(s,b)\right),
$$

$$
\mathcal L_{\mathrm{rank}}
=
\frac{1}{|\mathcal P_s|}
\sum_{(a,b)\in\mathcal P_s}
\log\left(
1+\exp\left[-y_{ab}\big(f_\theta(s,a)-f_\theta(s,b)\big)\right]
\right).
$$

$A_H(s,a)=A_H(s,b)$ 的 pair 跳过。主方法不用 regret head、softplus、$\beta$、selector-set best 或 pair magnitude 权重；这些最多作为非核心附录消融。

### 6.6 部署

在每个决策步：

$$
a_t
=
\arg\max_{a\in\mathcal A_C(s_t)}
f_\theta(s_t,a).
$$

使用 stable-key tie-break。对完整 $\mathcal A_C(s)$ 分块编码并做 streaming argmax；不采样 256 个候选，不在线 rollout，不调用 solver。

## 7. 实验设计

### 7.1 数据阶段

#### Diagnostic set

使用当前 10 个 Sats1/M100 instances 加上 Sats20 场景中的压力测试状态。只从 dev instances 抽取 100 个去重 prefix hashes，按以下维度分层：

- instance；
- observed action 是 `SKIP` / non-SKIP；
- timestep quartile；
- candidate-count quantile；
- objective；
- solution source family。

#### Paper set

只有 500-state gate 通过后才建立。目标是 EOS-Bench 中 3 个规模档、共 30–60 个独立 instances。每档先按 instance 固定 train/dev/test，strong solver 只在小规模或代表性子集运行；其余使用已有或廉价 heuristic schedules。不能为凑行数复制同一场景的 solver traces。

### 7.2 100-state diagnostic

主配置为 $B=16, H=5$。

1. 先运行 10-state smoke test，测量每次 force + continuation 的真实耗时；
2. 100 个状态全部运行 observed、SKIP 与结构化/通用 selectors 的 query union；
3. 至少 20 个 $|\mathcal A_C(s)|\le128$ 的状态做 exhaustive $Q_H$；
4. 至少 10 个小状态比较 objective-greedy、beam-8 和可解 MIP continuation；
5. 所有状态重放两次并比较 hashes；
6. 报告 return spread、双尾覆盖、worst-decile recall、NDCG、Spearman 和 query failure。

### 7.3 500-state selector gate

- 冻结表示、$H=5$ 与 selector；
- 比较 Uniform-K、Policy-sampling-K、Policy-top-K、generic feature farthest-first 和 CCOD；
- 所有方法使用完全相同的状态、候选集、$Q_H$、模型、pairwise loss、$K$、种子和训练步数；
- 在 dev instances 上训练小模型并跑完整 schedule；
- 若 selector 只提高离线 ranking、没有完整 schedule 趋势，则不扩到 2,000 states。

### 7.4 2,000-state training set

- 只保留 500-state gate 中最强的两个通用 selectors 与 CCOD；
- labels 按 `(state_hash,candidate_key)` 独立缓存；
- $B\in\{4,8\}$ 使用 $B=16$ 排序的前缀，不重复 rollout；
- labels、split、features、continuation、objective 和 seeds 全部冻结；
- 在 Mac 上先完成 32-state overfit、一个 seed 的 CPU pilot 和 learned-scheduler feasibility；
- 通过后才准备 3090 的紧凑训练包。

不默认扩展到全部 7,500 或 20,500 rows。是否继续扩展只由独立 instance 覆盖和学习曲线决定。

### 7.5 最低学习基线

#### Imitation baselines

- BC-Single；
- BC-All；
- group-normalized quality-weighted BC。

#### Cost-sensitive baselines

- SeaRNN-style Uniform-K；
- SeaRNN-style Policy-sampling-K；
- SeaRNN-style Top-K；
- generic feature farthest-first；
- Herrmann-style direct $Q_H$ regression；
- 小型 exhaustive states 上的 full-action LOLS/AggreVaTe-style cost-to-go 上界；
- CCOD footprint selector。

所有 sampled cost-vector 方法共享模型、label oracle、loss、预算和训练步骤，只替换 selector。

#### Scheduling references

- profit-first / completion-first / balanced greedy；
- MIP（只在可解规模）；
- SA / GA / ACO；
- PPO（仅在已有可复现 checkpoint/结果的场景）。

这些方法主要提供 schedule quality–runtime 参照，不要求在 3090 上重训。

### 7.6 主指标

#### 完整 schedule

- Total Profit；
- Task Completion Rate；
- Timeliness；
- Resource Balance；
- combined $F_\omega$；
- Feasibility Rate；
- online wall-clock 与 peak memory。

主指标是 held-out instance 上的 $F_\omega$ 差值；单步 top-1 accuracy 只作辅助诊断。

#### 查询与标签效率

- TailRecall@B；
- worst-decile regret recall@B；
- best-decile utility recall@B；
- NDCG / Spearman；
- 每 1,000 次 counterfactual queries 的完整 schedule $F_\omega$ 增益；
- Mac CPU label wall-time；
- cache hit、failure 和 duplicate-prefix rate。

#### 统计

- final shortlisted methods 使用 5 个训练 seeds；
- paired comparison 以同一 held-out instance 和 objective 为配对；
- cluster bootstrap 以 instance 为重采样单位；
- 报告 95% confidence interval 与 effect size；
- feasibility 单独报告，不能用 objective 增益掩盖不可行 schedule。

### 7.7 最小消融

只保留能回答核心问题的消融：

1. $B\in\{4,8,16\}$；
2. $H=5$ 为主，$H=10$ 只在小诊断集检查稳定性；
3. 去掉 conflict footprint；
4. 去掉 resource footprint；
5. footprint-permutation 主负对照；
6. 独立 MLP 与集合条件 scorer；
7. direct $Q_H$ regression 与 pairwise ranking。

删除旧方案中的 $K\in\{32,64\}$、$H=20$、$\beta$ 大网格、双头权重网格和 learner-state 主实验。

## 8. 单 Mac + 按需 RTX 3090 的计算方案

### 8.1 当前真实设备

- MacBook Air，Apple M4，10-core，24 GB 内存；
- 本地无 CUDA；
- 已安装并固定 Python 3.10.20，项目使用本地 `.venv`；

- `data/` 约 26 GiB，不能整体载入内存；
- 当前 BC 约 19K 参数，20,500 rows 的 CPU 训练约 38–40 秒/epoch。

因此默认把 Mac 当作 CPU 开发、标签与评估机器；方案不依赖 MPS 恢复。

### 8.2 Mac 负责的工作

- Python 3.10 可复现环境与依赖锁；
- 数据 inventory、instance split 和 prefix 去重；
- 状态重放、约束测试和目标一致性；
- footprint 计算；
- 10 → 100 → 500 → 2,000 states 的 CPU counterfactual labeling；
- label cache、checkpoint/resume 与 checksum；
- CPU overfit / one-seed pilot；
- learned scheduler 的完整 schedule rollout；
- 统计分析、表格和作图。

长时间 CPU labeling 默认从 2 workers 开始，稳定后最多尝试 4 workers；每个 state 独立落盘，避免 Mac 长时间热降频或中断导致整批重跑。

### 8.3 存储与数据流

- 不再生成 `all_shuffled.jsonl` 或新的整库 concat 副本；
- 用 instance/schedule 索引 manifest 代替物理 shuffle；
- 原始 scenario 与 prefix keys 只保留一份；
- numeric features、mask、pair labels 单独分片；
- cache key 为：

$$
(\text{state hash},\text{candidate key},H,
\text{continuation hash},\text{objective hash},
\text{constraint-config hash}).
$$

- 每个 shard 写临时文件后原子 rename，并保存 row count 与 checksum；
- 训练 3090 只上传冻结后的 numeric shards、代码、配置和 manifest，目标传输量小于 1 GiB；
- 不上传 26 GiB 的重复 JSONL，也不让 GPU 等待 raw JSON parsing。

### 8.4 何时才允许租 3090

以下全部满足后才租卡：

1. P0 全部通过；
2. 100-state diagnostic 证明有稳定的机会代价信号；
3. 500-state selector gate 证明 footprint 查询优于通用 selector；
4. 2,000-state labels 完成并 checksummed；
5. scenario split、feature schema、$B$、$H$、continuation 和 seeds 已冻结；
6. Mac 上 32-state overfit 通过；
7. learned scheduler 在 held-out small instances 上 100% feasible；
8. 训练包在 CPU 上完整跑通一个 epoch；
9. 环境 lock、启动命令、预计 epoch 数和 early-stop 条件已记录。

### 8.5 3090 使用方式

3090 只用于模型训练，不用于：

- 生成 counterfactual labels；
- 跑 MIP / SA / GA / ACO；
- 状态重放；
- 最终 schedule evaluation；
- 临时清洗 26 GiB JSONL。

建议一次短租分三段：

1. **30 分钟 smoke**：一个 seed、一个 epoch；出现 NaN/OOM、GPU 利用率长期低于约 30–40%、数据等待大于计算时间，立即停止；
2. **1–2 小时 pilot**：enriched BC 与 CCOD 各一个 seed；held-out ranking 不为正或 decoder 失败，立即停止；
3. **4–8 小时正式 block**：只跑入选模型、5 seeds 和最小消融；每 epoch checkpoint，连续 2–3 个 epoch 无改进即 early stop。

租用结束前下载 checkpoint、optimizer state、训练日志、配置、环境清单和 checksums。完整 schedule rollout 回 Mac 执行。

由于模型目标小于 0.5M 参数，如果缓存后的 Mac CPU 训练已经足够快，允许完全不租 3090；GPU 是缩短正式多 seed 训练的工具，不是论文成立条件。

## 10. Go / No-Go 标准

| Gate | Go 条件 | 失败后的动作 |
|---|---|---|
| P0 correctness | 两次重放 hashes 100% 一致；0 个不可行 force；目标增量一致；instance split 无泄漏 | 不做标签，不训练 |
| Label validity | 小型 exhaustive 子集上 cheap continuation 与 beam-8/MIP 的 median Spearman ≥ 0.7 | 修 continuation / 增大 `H`，不扩数据 |
| Signal | 至少预注册比例的 states 有非平凡 $Q_H$ spread，且 BC top candidates 中存在可测的高代价动作 | 研究动机 No-Go |
| Selector | CCOD 相对最佳 Uniform/Policy/Top-K 的 TailRecall@16 至少 +10 个百分点，paired-bootstrap 95% CI 下界 > 0 | 方法性新颖性 No-Go |
| Learnability | scenario-held-out pairwise accuracy/AUC 明显高于 BC score 与随机，Spearman CI > 0 | 修 representation；一次后仍失败则 No-Go |
| 500-state closed loop | 完整 schedule $F_\omega$ 有一致正趋势且 feasibility 不下降 | 不扩到 2,000 |
| Final method | 5 seeds、held-out instances 上，CCOD 相对最佳 sampled cost-vector baseline 的 $F_\omega$ 差值 95% CI > 0；feasibility 不下降；online RT $\le$ BC 的 1.2 倍 | 论文方法 No-Go，最多写诊断/benchmark paper |
| Negative control | footprint permutation 使 TailRecall 和下游增益回落到 uniform 区间 | 若不回落，撤回 footprint 因果解释 |

“非平凡 spread”的数值阈值要在看到 diagnostic label 前，根据统一归一化 $F_\omega$ 预注册；建议从 state 内 $P_{90}-P_{10}\ge 0.01$ 开始，并同时报告连续 effect，而不是只报通过率。

## 11. 主要风险与应对

### 风险 1：cheap continuation 排序不可信

应对：小状态 exhaustive + beam-8/MIP rank stability；失败时只修 label oracle，不训练更大模型。

### 风险 2：footprint 只是通用 diversity 的换名

应对：加入 generic feature farthest-first、公平 SeaRNN-style selectors 和 selector-only permutation；贡献只由同预算增益支撑。

### 风险 3：现有 trace 跨实例覆盖不足

应对：100/500-state 只做诊断；方法通过后才建立 30–60 instance 的紧凑多规模 paper set。

### 风险 4：off-trace distribution shift

应对：最终评估必须从空 schedule closed-loop 运行。若 only-trace 学习的收益在早期消失，记录 state drift；只有主方法已经有正信号时，才考虑小规模 learner-state augmentation，不把它预设为主贡献。

### 风险 5：候选 cap 改变最优动作

应对：始终写 $\mathcal A_C$；在小实例上做 cap stability；正式比较中所有方法共享 $C$。

### 风险 6：标签生成成为真正瓶颈

应对：100 → 500 → 2,000 gate、prefix 去重、query union 缓存、$B$ 前缀复用、2–4 workers、每 state checkpoint。禁止直接运行旧方案的 $7{,}500\times64\times20$。

### 风险 7：GPU 实际无加速

应对：先制作 numeric shards；30 分钟 smoke 有明确 kill switch；若 GPU 等 I/O 或相对 Mac 无实质加速，立即释放。

## 12. 可辩护贡献

若所有 Go 条件通过，贡献写成以下三条：

1. **问题与数据协议**：将 heterogeneous solver final schedules 明确定义为 canonical solution-induced constructive traces，并建立精确重放、instance-level split 和 capped dynamic action set 上的 partial cost-vector supervision 协议；
2. **方法**：提出 constraint-consistent 的 EOS conflict/resource footprint query allocation，在固定反事实预算下提高 $Q_H$ 双尾覆盖与排序监督效率；
3. **实证**：在 EOS-Bench 上联合评估完整 schedule quality、online runtime 与 Mac CPU offline labeling cost，并通过 exhaustive small states、strong-continuation rank check 和 selector-only permutation 证伪协议验证机制。

双头网络、regret penalty decoder、learner-state aggregation 和“首次反事实蒸馏”均不写入贡献。

- strong continuation rank stability；

- 在线 runtime 与离线 query cost 同时报；
