## 摘要（实验前版本）

对地观测卫星调度需要在随状态动态变化的“任务—卫星—观测窗口”候选集中进行长序列决策；一个局部动作会通过姿态转移、窗口冲突与星上资源占用改变后续可行域。异构求解器可产生不同质量的可行最终解，但将其规范重放为构造轨迹时只暴露被执行动作，普通行为克隆因而无法区分未选可行动作之间不同的下游机会代价。本文提出冲突覆盖机会代价蒸馏方法 CCOD。首先，将异构求解器输出转换为可哈希、可精确重放的 solution-induced constructive traces，并固定目标函数、候选上限与稳定动作顺序。其次，在每个状态仅查询上限为 $B$ 的候选动作，采用确定性的有界视域 continuation 估计短期回报；同时利用未来窗口删除、优先级质量损失、最后可行机会、姿态转移余量与资源压力构造 EOS conflict/resource footprint，据此分配反事实查询预算。最后，使用已查询动作对的相对回报训练集合条件的单头评分器；部署时对完整 capped feasible set 一次打分，不执行 rollout、树搜索或在线求解器调用。实验将基于 EOS-Bench 按实例划分数据，在相同查询上限与 continuation 下比较 uniform、policy、top-k 和通用多样性选择器，并联合报告容量平衡的双尾覆盖、排序质量、完整排程目标、可行率、在线时延与离线标注成本。正式实验尚未完成，因此本摘要不预报效果数值；方法结论受预先定义的 100-state、exhaustive-selector 与 500-state Go/No-Go 门槛约束。

**关键词：** 对地观测卫星调度；反事实监督；机会代价；模仿学习；查询效率；排序蒸馏

## Abstract (proposal version; results pending)

Earth-observation satellite scheduling requires decisions over large, dynamic, and strongly constrained candidate sets while balancing profit, completion, timeliness, and resource utilization. Feasible final schedules of heterogeneous quality reveal only selected actions after canonical reconstruction and therefore do not expose the downstream opportunity costs of feasible alternatives. We propose **Conflict-Covered Opportunity-Cost Distillation (CCOD)**, a resource-aware offline learning framework that reconstructs exact scheduling prefixes, allocates a counterfactual query budget with fixed upper bound $B$ using EOS conflict and resource footprints, estimates action values with deterministic finite-horizon continuations under a unified objective, and distills the resulting pairwise order into a lightweight set-conditioned policy with search-free deployment. The planned evaluation uses instance-level splits on EOS-Bench and jointly measures label validity, capacity-balanced tail coverage, complete-schedule quality, feasibility, online runtime, and Mac-CPU labeling cost. No empirical claim is inserted before the preregistered diagnostic and held-out evaluation gates are passed.

## 主要创新与声明边界

以下三项是待实验验证的论文主张；只有对应 gate 通过后，才改为确定语气。

1. **面向 selected-only solver schedules 的可复现监督协议。** 将异构求解器最终排程明确界定为 canonical solution-induced constructive traces，而非求解器真实内部轨迹；通过精确前缀重放、稳定动作键、capped dynamic action set、统一目标和 instance-level split，把终局排程转化为可复核的 partial cost-vector supervision。该项是 EOS 场景的数据语义与实验协议，不声称 exact replay 本身是通用算法创新。
2. **约束一致的 EOS 结构感知查询分配。** 提出 conflict/resource footprint，用候选动作导致的未来窗口删除、优先级质量损失、最后机会损失、姿态与窗口余量、星上资源瓶颈刻画其潜在机会代价；在固定上限 $B$ 下联合覆盖高风险端、低风险端、结构多样性及少量 seeded-uniform 支持。核心主张不是 farthest-first 或主动采样本身，而是 EOS 约束一致的 footprint 能否提高单位反事实查询的容量平衡双尾覆盖和排序监督效率。
3. **带完整成本核算与证伪机制的离线机会代价蒸馏闭环。** 在精确状态上以固定上限 $B$ 生成 bounded-continuation 相对排序标签，训练集合条件的单头评分器，并在完整 capped feasible set 上实现无搜索推理；同时用 exhaustive small states、strong-continuation rank check、selector-only footprint permutation 和预算配对实验，把“标签质量—离线计算—完整排程收益—在线时延”置于同一评价闭环。这里的贡献是 EOS selected-only 数据条件下的组合与验证，不声称 cost-to-go、pairwise ranking 或 search-free distillation 是新的。

新颖性核验的保守结论为 **Level 2 — High Overlap**：SeaRNN、LOLS、AggreVaTe/AggreVaTeD 已覆盖通用 cost-to-go/rollout supervision，Herrmann–Schaub 已覆盖 EOS 中从 MCTS 动作价值到无搜索策略的蒸馏 [3, 7–10]。因此论文的成立点必须是上面三项的精确组合及其可测收益，而不是泛称“反事实学习”或“regret distillation”。

## 0. 一页结论

这篇论文不再声称“首次使用替代动作 rollout、cost-to-go、regret 或无搜索蒸馏”。SeaRNN、LOLS、AggreVaTe/AggreVaTeD 和 EOS 调度中的 MCTS-Q 蒸馏已经覆盖了这些一般机制 [3, 7–10]。

论文唯一需要负重的方法性主张是：

> 在 EOS-Bench 的动态大候选动作集中，候选动作具有可由冲突图、窗口稀缺度、姿态转移余量和星上资源瓶颈描述的结构。本文待验证的核心命题是：用该结构分配每状态至多 $B$ 次反事实查询，能否比 uniform、policy sampling、top-k 与通用多样性选择更高效地覆盖机会代价两端，并把这种排序知识蒸馏为部署时不做搜索的轻量策略。

最小方法闭环是：

1. 将异构 solver 的最终 schedule 规范化为可确定性重放的 **solution-induced constructive traces**；
2. 在精确前缀上，对每个状态只查询至多 $B=16$ 个动作；
3. 用 EOS conflict/resource footprint 选择这些动作；
4. 用统一目标下的 bounded continuation 得到 signed advantage；
5. 用单头、集合条件的 scorer 学习 queried pairs 的排序；
6. 部署时对枚举出的全部合法动作一次前向打分并取 `argmax`，不做 rollout、MCTS 或在线 solver 调用。

当前最重要的工作不是租 GPU。`force_action`、$H=5$ continuation、6/2/2 instance split、无标签 100-state query plan 与可恢复单 worker runner 已经实现。`aab588e` 下的第一个冻结 state 已真实完成 16/16 查询及零新增查询的热恢复；该试运行促成 `c52ef85` 对父锁租约、坏缓存粘性和不可覆盖证据链的进一步加固。下一步是用当前提交在新 run-dir 复跑同一 state，再逐 state 完成 1,570 条查询；现阶段 1/100 只证明执行闭环，不构成机会代价信号或方法收益。现有 BC 只有约 19K 参数，Mac CPU 每个 epoch 约 38–40 秒，尚不存在 GPU 瓶颈。

## 1. 论文定位、边界与目标读者

### 1.1 推荐定位

目标是航天调度、运筹优化、智能优化或应用型 AI 论文，而不是把它包装成通用序列学习新理论。

建议叙事顺序：

1. EOS 动作空间大、动态变化且强约束；
2. solution-induced trace 只告诉模型“做了什么”，没有告诉模型“其他动作会损失什么”；
3. 对所有动作做 continuation 太贵；
4. EOS 的冲突/资源结构可以帮助选择最值得查询的动作；
5. 结构化有限查询能以较低离线代价改善完整 schedule，同时保持在线推理轻量。

### 1.2 核心研究问题

> 如何从 heterogeneous solver final schedules 构造的规范轨迹中，以极少的 EOS 结构感知查询恢复未执行动作的机会代价排序，并将其蒸馏为无需在线搜索的卫星调度策略？

### 1.3 明确不声称的内容

- 不声称 forced alternative action rollout 是新的；
- 不声称 local regret / cost-to-go imitation 是新的；
- 不声称 random、policy sampling、top-k、uncertainty 或 generic diversity sampling 是新的；
- 不把双头网络、softplus regret、$z-\beta r$ 或某个损失组合列为核心贡献；
- 不把 MIP、SA、GA、ACO 的 final schedule 误称为其内部逐步决策轨迹；
- 不在主方法中加入 learner-state aggregation；它只在主结果已经为正、且 off-trace shift 被证实后作为后续扩展；
- 不承诺跑完 EOS-Bench 全部实例，也不把 JSONL 行数当作独立样本数。

### 1.4 相关工作与创新边界

#### EOS 调度与学习型求解

敏捷 EOS 排程的经典问题定义与二十年方法综述分别见 Lemaître 等 [12] 和 Wang 等 [11]。EOS-Bench 统一了多种敏捷/非敏捷场景、约束、求解器和五维评测指标，是本文的任务与复现基座 [1]。既有学习方法已经覆盖多个相邻方向：Hadj-Salah 等用深度强化学习处理简化的观测获取调度 [2]；Herrmann–Schaub 用 MCTS 生成状态–动作价值并回归为可在星上快速执行的无搜索价值网络 [3]，并进一步研究敏捷 EOS 的强化学习调度 [13]；Jacquet 等用 GNN/PPO 表示大规模候选冲突，并在部署阶段继续使用 MCTS 改善解 [4]；近期工作还研究了跨规模泛化的资源约束 DRL [5]，以及以动作冲突图、GAT 与 DQN 选择单星目标—成像时刻 [14]。Belaid 的 CCA 则在“目标已知、操作约束未知”的 EOS 设定下，主动查询二值可行性 oracle 以获取约束 [16]；这是同域的主动查询边界，但其查询对象不是 CCOD 中已知约束下的备选动作截断回报。因此本文不能声称“首次把学习用于 EOS 调度”“首次蒸馏搜索价值”“首次使用图动作表示”“首次处理动态候选集”或“首次在 EOS 中主动查询”。

#### Cost-to-go imitation 与 learning to search

DAgger 说明了行为克隆在 learner-induced state distribution 上的复合误差问题 [6]；AggreVaTe 进一步用专家 cost-to-go 区分不同动作错误的后果 [7]。LOLS 对合法替代动作执行 rollout，并以 rollout loss 相对最佳动作的差定义局部代价 [8]；Deeply AggreVaTeD 将训练期动作价值用于可微神经策略 [9]；SeaRNN 则在大词表结构预测中采用 forced alternatives、完成式 rollout、局部 cost vector 和 uniform/policy/top-k 子采样，同时保持测试时不做 rollout [10]。Goodman 等还在模仿学习中研究了 targeted exploration，进一步说明“非均匀选择值得探索的动作”本身已有先例 [15]。这些工作已覆盖 CCOD 的通用学习骨架。

#### 本文相对差异

本文的可辩护 delta 不是单个通用算子，而是：从只暴露最终 assignment 集合的异构 EOS solver 解中构造可验证状态；在超大、动态、受真实约束过滤的 task–satellite–window 候选集上，用 EOS conflict/resource footprint 分配固定上限的反事实预算；以标签成本、容量平衡双尾覆盖和完整 schedule 收益共同检验该分配是否优于 SeaRNN-style 通用采样。最接近的单篇方法威胁是 SeaRNN，最接近的局部 regret 目标是 LOLS，最接近的同域价值蒸馏工作是 Herrmann–Schaub [3, 8, 10]。只有这个组合在公平预算下带来预注册门槛要求的收益，CCOD 的方法贡献才成立。

## 3. 可证伪假设

### H1：机会代价异质性

在相同精确状态下，不同合法动作的 $Q_H(s,a)$ 存在有实际意义的差异；否则本研究关于 bounded-horizon 机会代价监督必要性的动机不获支持。

### H2：结构可预测性

候选对应卫星的资源状态、窗口稀缺度、姿态转移余量和未来冲突 footprint 能预测动作间 $Q_H$ 排序；否则昂贵标签无法被轻量模型泛化。

### H3：查询效率

在相同查询上限 $B$、相同 continuation、相同模型、相同训练步数下，CCOD 的 footprint selector 比最佳 uniform/policy/top-k/generic-diversity selector 覆盖更多真实高代价与低代价动作。

### H4：完整调度收益

CCOD 学到的单头 scorer 在 held-out instances 上提高完整 schedule 的 $F_\omega$，同时 feasibility 不下降、在线推理仍接近 BC。

### 3.1 唯一负重变量与负对照

对预注册的 $17\le C_s=|\mathcal A_C(s)|\le128$ exhaustive dev states，令查询集合为 $U_B(s)$，并定义：

$$
\begin{aligned}
q_s&=\max\{1,\lceil0.1C_s\rceil\},\\
b_s&=|U_B(s)|,\\
m_s&=\min\{q_s,\lfloor b_s/2\rfloor\}.
\end{aligned}
$$

按 stable candidate key 打破 $Q_H$ 并列，记真实得分最高和最低的 $q_s$ 个动作为 $\operatorname{Best}_{q_s}(s)$ 与 $\operatorname{Worst}_{q_s}(s)$。容量平衡双尾覆盖定义为：

$$
\operatorname{BTailCov}_{B}(s)
=\frac{1}{2}
\left[
\frac{\min\{|U_B(s)\cap\operatorname{Best}_{q_s}(s)|,m_s\}}{m_s}
+
\frac{\min\{|U_B(s)\cap\operatorname{Worst}_{q_s}(s)|,m_s\}}{m_s}
\right].
$$

主证伪统计量为 $V_{16}=|\mathcal S_{\mathrm{exh}}|^{-1}\sum_{s\in\mathcal S_{\mathrm{exh}}}\operatorname{BTailCov}_{16}(s)$。$m_s$ 将有限预算平均分配给两端，使分数在不同候选规模下仍位于 $[0,1]$；只看最差动作会忽略“是否查询到好动作”。该指标依赖全部动作的 $Q_H$，因此只用于 20 个 exhaustive dev states，不在非穷举的 500-state 工程集上计算。

唯一主负对照是：

> 在每个状态内随机置换 $v_s(a)$ 与 candidate key 的映射，且只在 selector 内置换；模型仍接收真实特征，$B$、随机种子、真实 continuation 标签、网络和优化步骤全部不变。

若置换后 $V_{16}$、NDCG 和每 1,000 次查询带来的完整 schedule 增益没有回落到 uniform 区间，则“EOS footprint 改善查询效率”的解释被证伪。

## 4. 问题定义

### 4.1 状态、动作与 cap

在状态 $s_t$，数学上的全部合法动作集合记为：

$$
\mathcal A(s_t).
$$

实际候选生成器受 `max_candidates` 上限 $C$、时间离散和枚举顺序影响，论文真正使用的集合记为：

$$
\mathcal A_C(s_t)\subseteq \mathcal A(s_t).
$$

`SKIP` 是普通合法动作。论文所有“完整动作集”表述均指完整的 $\mathcal A_C(s)$，不得把 cap 后集合写成理论上的 $\mathcal A(s)$。

要求：

- cap 必须记录在每个 run manifest；
- candidate key 与 tie-break 顺序稳定；
- trace action 不在 $\mathcal A_C(s)$ 时不得在 validation/test 中偷偷注入；
- 小实例上比较多个 $C$，确认 cap 不改变主要参考排序后再冻结 $C$。

### 4.2 规范构造轨迹

给定 final schedule $Y$，固定：

1. task priority 降序；
2. priority 相同时按稳定 task id；
3. 若 $Y$ 为当前 task 分配 assignment，则重放该 candidate key；
4. 否则执行 `SKIP`。

由此得到：

$$
\tau(Y)=\{(s_t,a_t^{\mathrm{obs}})\}_{t=1}^{T}.
$$

$a_t^{\mathrm{obs}}$ 表示 trace observed action，不写成求解器真实在线动作。

### 4.3 统一目标

最终目标使用 EOS-Bench 的归一化多指标组合：

$$
F_\omega(S)
=
\omega_p\,\mathrm{Profit}(S)
+\omega_c\,\mathrm{Completion}(S)
+\omega_t\,[1-\mathrm{TM}(S)]
+\omega_b\,\mathrm{Balance}(S),
$$

其中：

$$
\omega_p+\omega_c+\omega_t+\omega_b=1,
\qquad
\omega_i\ge 0\quad(i\in\{p,c,t,b\}).
$$

其中 EOS-Bench 原始 timeliness metric $\mathrm{TM}\in[0,1]$ 越小越好，故最大化目标使用 $1-\mathrm{TM}$。所有 trace 质量、counterfactual label 和最终评估必须调用同一 `ObjectiveModel` 和同一 $\omega$。不再给 `SKIP` 额外添加 $-1$，也不再把 $10\times\Delta F$ 与程序性惩罚混合。

主实验先冻结 balanced objective；profit-first 和 completion-first 只作为较小的目标条件泛化实验。模型输入显式包含 $\omega$。

## 6. 方法：CCOD

### 6.1 精确 counterfactual continuation

对状态 $s$ 和被查询动作 $a\in\mathcal A_C(s)$：

1. deep-copy 精确 partial schedule；
2. 强制执行 `a`；`SKIP` 保持当前任务未分配；
3. 使用固定的 deterministic objective-greedy policy 继续到总共 $H$ 个决策；强制动作计作第 1 步，随后最多执行 $H-1$ 个 greedy 决策；
4. 所有 ties 用 stable candidate key 打破。

主配置：

$$
H=5.
$$

令 $S(s)$ 表示状态 $s$ 对应的 partial schedule，令 $\mathcal C_H(s\oplus a)$ 表示先强制执行 $a$、再按上述策略完成总计至多 $H$ 个决策后得到的 schedule。若剩余任务少于 $H$，则在任务耗尽时提前终止。定义：

$$
\begin{aligned}
Q_H(s,a)
&=F_\omega\!\left(\mathcal C_H(s\oplus a)\right)
-F_\omega\!\left(S(s)\right),\\
A_H(s,a)
&=Q_H(s,a)-Q_H\!\left(s,a^{\mathrm{obs}}\right).
\end{aligned}
$$

其中 $A_H$ 是相对 observed action 的 signed advantage。$Q_H$ 只是冻结 continuation 下的 $H$ 步截断回报代理，不是全局最优 cost-to-go；正文与图表均不得省略这一限定。

这里没有 budget-set best reference、独立 `SKIP` penalty 或 selector-dependent normalization。同一状态的 pairwise 差值为：

$$
A_H(s,a)-A_H(s,b)=Q_H(s,a)-Q_H(s,b).
$$

只在完整枚举的小型诊断状态上，定义用于评估的非负机会代价：

$$
\operatorname{OC}_H(s,a)
=
\left[\max_{b\in\mathcal A_C(s)}Q_H(s,b)\right]-Q_H(s,a).
$$

$\operatorname{OC}_H$ 表示在已冻结 capped action set $\mathcal A_C(s)$ 上定义的 opportunity cost；它会随候选上限 $C$ 改变，因此 cap 必须进入 run manifest。该量只用于定义 high-cost/worst-decile coverage，不作为规模化训练时依赖“预算集合内最大值”的标签。

在小型 exhaustive dev 子集上，再用 beam-8 或可解的 MIP continuation 检查 cheap continuation 的排序稳定性。主方法不在线调用它们。

Beam-8 仅用于预注册的 10 个 exhaustive dev 状态。对每个被强制动作 $a$，深度 1 的唯一节点为 $s\oplus a$；随后每层枚举各节点完整的 $\mathcal A_C(\cdot)$，直到总决策深度 $H$。节点分数为当前 partial schedule 的 $F_\omega$，每层保留分数最高的 8 个节点；平局按完整 ActionKey 序列的字典序打破。任务耗尽节点保留为 terminal，不再扩展。定义：

$$
Q_H^{\mathrm{beam8}}(s,a)
=
\max_{S'\in\mathcal B_H(s,a)}F_\omega(S')
-F_\omega(S(s)),
$$

其中 $\mathcal B_H(s,a)$ 是深度 $H$ 或提前耗尽时的 beam frontier。在每个 exhaustive 状态上，对全部 canonical actions 比较 $Q_H^{\mathrm{greedy}}$ 与 $Q_H^{\mathrm{beam8}}$ 的 Spearman 相关。任一向量为常数、包含非有限值或相关系数未定义时，该状态以及整个 Label-validity gate 均判失败，不得从 median 分母删除。

### 6.2 EOS conflict/resource footprint

对候选动作 $a$，构造无 counterfactual rollout 的结构签名：

$$
v_s(a)=
\left[
\begin{array}{c}
\text{immediate normalized gain}\\
\text{deleted future-window count}\\
\text{deleted future priority mass}\\
\text{last-feasible-window loss count}\\
\text{remaining storage / power ratios}\\
\text{storage / power consumption ratios}\\
\text{transition slack}\\
\text{window slack}\\
\text{same-satellite contention}\\
\text{opportunity scarcity}
\end{array}
\right].
$$

令 $\mathcal O_L(s)$ 为规范任务顺序中后续至多 $L$ 个任务在状态 $s$ 下的可行 observation opportunities。未来冲突集合定义为：

$$
\mathcal D_L(s,a)
=
\left\{o\in\mathcal O_L(s):
o\text{ 在 }s\text{ 下可行，且在 }s\oplus a\text{ 下不可行}\right\},
$$

其中只扫描后续固定 $L$ 个任务的 opportunities，删除关系由与 rollout 相同的 constraint checker 判断。$L$ 只根据 train split 的无标签规模/耗时 profiler 冻结，并且早于任何 $Q_H$ 查询；默认候选值为 32，不得根据 dev 标签或 selector 结果调整。该计算可用时间区间与 satellite 索引缓存，不能退化成全局 $O(|\mathcal A|^2)$ 暴力比较。

对后续任务 $m$，令 $\mathcal O_m(s)\subseteq\mathcal O_L(s)$ 为其在 $s$ 下的可行 opportunities，并令：

$$
\mathcal M_L(s)=\left\{m:\left|\mathcal O_m(s)\right|>0\right\}.
$$

结构风险仅使用优先级加权的机会删除比例：

$$
r_s(a)
=
\frac{
\displaystyle
\sum_{m\in\mathcal M_L(s)}
p_m
\frac{\left|\mathcal D_L(s,a)\cap\mathcal O_m(s)\right|}
{\left|\mathcal O_m(s)\right|}
}{
\displaystyle
\sum_{m\in\mathcal M_L(s)}p_m
}.
$$

若 $\mathcal M_L(s)=\varnothing$ 或分母为 0，则定义 $r_s(a)=0$；`SKIP` 的结构风险固定为 0。资源比例、资源消耗、transition/window slack 仍属于 $v_s(a)$，但只参与 farthest-first 距离，不进入风险最大/最小端点，避免通过未冻结权重改变风险语义。

签名各维只用 train split 统计量做 robust scaling；不根据 counterfactual label 调 selector 权重。

### 6.3 Conflict-covered query allocation

selector 输出一个稳定、可取前缀的候选排序。主预算：

$$
B=16.
$$

selector 先构造长度至多 16 的稳定序列，槽位从 1 开始计数：

1. slot 1：$a^{\mathrm{obs}}$；
2. slot 2：合法 `SKIP`；
3. slot 3：尚未选择动作中 $r_s(a)$ 最大者；
4. slots 4、8、12、16：按 `hash(seed,state_hash,ActionKey)` 排序的 seeded-uniform 动作；
5. slot 5：尚未选择动作中 $r_s(a)$ 最小者；
6. 其余槽位：在 robust-scaled $v_s(a)$ 空间做 deterministic farthest-first。

所有平局使用 stable ActionKey。若某一角色与已选动作重复或当前状态不存在该角色，则从该角色的稳定候选序列取首个未选动作；仍无可用动作时，使用 stable-uniform 首个未选动作补位。保留槽位编号，不因重复而移动后续 uniform 槽位。$B\in\{4,8,16\}$ 始终取同一序列前缀；当 $C_s<B$ 时只返回全部唯一候选，使预算消融可复用标签。

该 selector 的贡献不是 farthest-first 算法本身，而是 EOS constraint-consistent footprint、风险两端覆盖和固定查询上限下的可证伪 query-efficiency。

### 6.4 单头集合条件 scorer

所有学习方法共享同一个轻量模型：

1. candidate encoder 编码 task、window、platform/sensor、资源和 footprint；
2. per-satellite encoder 聚合当前卫星的 assignments 与剩余资源；
3. DeepSets 用 mean + max pooling 得到候选集合上下文；
4. 单头 scorer 输出：

$$
f_\theta(s,a)\in\mathbb R.
$$

模型参数目标小于 0.5M。不同卫星候选通过其 per-satellite context 区分，不依赖任意 satellite id 的记忆。

### 6.5 排序蒸馏

先用 BC 初始化 scorer。令 $\epsilon_{\mathrm{pair}}\ge0$ 为数值复现审计后、首个 $Q_H$ 标签产生前写入配置并哈希冻结的容差；不得根据标签分布调节。定义：

$$
\mathcal P_s
=
\left\{
(a,b):a,b\in U_B(s),\ a\ne b,\
\operatorname{key}(a)<\operatorname{key}(b),\
\left|A_H(s,a)-A_H(s,b)\right|>\epsilon_{\mathrm{pair}}
\right\}.
$$

每对按 stable ActionKey 固定方向，并令：

$$
y_{ab}=\operatorname{sign}\!\left(A_H(s,a)-A_H(s,b)\right).
$$

状态损失采用显式分段定义：

$$
\ell_s=
\begin{cases}
\displaystyle
\frac{1}{|\mathcal P_s|}
\sum_{(a,b)\in\mathcal P_s}
\log\!\left(
1+\exp\!\left[
-y_{ab}\bigl(f_\theta(s,a)-f_\theta(s,b)\bigr)
\right]
\right),
& |\mathcal P_s|>0,\\
0,
& |\mathcal P_s|=0.
\end{cases}
$$

令 $\mathcal S_+=\{s:|\mathcal P_s|>0\}$，则：

$$
\mathcal L_{\mathrm{rank}}
=
\frac{1}{|\mathcal S_+|}
\sum_{s\in\mathcal S_+}\ell_s.
$$

空 pair 状态的损失记为 0，但不进入平均分母；若一个批次 $\mathcal S_+=\varnothing$，跳过该优化步并记录计数。主方法不用 regret head、softplus、$\beta$、selector-set best 或 pair magnitude 权重；这些最多作为非核心附录消融。

### 6.6 部署

在每个决策步：

$$
a_t
=
\arg\max_{a\in\mathcal A_C(s_t)}
f_\theta(s_t,a).
$$

使用 stable-key tie-break。对完整 $\mathcal A_C(s)$ 分块编码并做 streaming argmax；不采样 256 个候选，不在线 rollout，不调用 solver。

## 7. 实验设计

### 7.1 数据阶段

#### Diagnostic set

划分使用 2.3 节已冻结的 6/2/2 instance split。100-state diagnostic **只使用** dev 的 `cities_08` 与 `cities_04`，每个 instance 固定抽取 50 个状态。首轮只使用 balanced objective，solution source 只包含 SA、GA 与 ACO；不混入 profit-first、completion-first 或 MIP source，避免在一个 100-state 小诊断中同时改变多个因素。

状态先在同一 instance 内按 `state_hash` 去重，合并其 source provenance；canonical source 只由 `seed`、`state_hash` 与预注册 solver family 坐标确定，并由该来源唯一确定 canonical observed ActionKey。随后使用只依赖标签前信息的稳定哈希排名抽样。一个状态只有在以下条件全部满足时才是 actionable：

- exact replay 与 state/candidate hashes 验证通过；
- `step <= 95`，为 $H=5$ 保留完整的诊断视野；
- 令 $C_s=|\mathcal A_C(s)|$，要求 $C_s\ge2$；
- canonical source 的唯一 observed ActionKey 是当前 capped feasible set 的成员，且 query plan 可确定性重建。

标签前 catalog 审计发现 124 个 canonical observed-SKIP 状态全部满足 $C_s=1$，与 actionable 条件 $C_s\ge2$ 互斥。因此首轮状态抽样的 observed-SKIP 目标在查看 $Q_H$ 前固定为 0；这不删除 SKIP 对照，因为每个入选状态的 query plan 仍强制包含一次当前任务的 `SKIP`。

最终目录必须是每个 dev instance 50 个、共 100 个唯一 actionable `state_hash`。其中以 $C_s\ge10$ 定义 signal-eligible，两个 dev instances 合计至少 80 个；该条件必须在发出任何 continuation query 前由 catalog 验证。不足 80 个时停止并报告覆盖不足，不根据已看到的 $Q_H$ 替换状态。为保证标签有效性核验可执行，selection 还硬性保证每个 dev instance 至少 10 个状态满足 $17\le C_s\le128$；随后 exhaustive 子集按 instance 各取 10 个，strong 子集再从 exhaustive 内部按 instance 各取 5 个，二者是嵌套关系而不是两批不同状态。

来源库存冻结在 `diagnostic_v1_sources.json`：十个场景同时记录 raw 与 semantic SHA-256，六个 dev balanced schedules 逐文件记录 SHA-256；库存文件哈希为 `sha256:27cb6a28fd94f7c0360f6dd351c1607f80424b92c5001efba8682b89915311c5`。sealed test 的场景只用于验证 instance identity 与 split 完整性；其 schedule、trace 和标签在方法锁定前均不读取。

Sats20 仅用于候选规模、内存与 wall-time 压力测试，不进入这 100 个状态，不进入 signal Go/No-Go 的分子或分母，也不被当作额外独立 instance。

#### Paper set

只有 500-state gate 通过后才建立。目标是 EOS-Bench 中 3 个规模档、共 30–60 个独立 instances。每档先按 instance 固定 train/dev/test，strong solver 只在小规模或代表性子集运行；其余使用已有或廉价 heuristic schedules。不能为凑行数复制同一场景的 solver traces。

### 7.2 100-state diagnostic

主配置为 $B=16, H=5$。

1. 10-state 真实 smoke 已完成；保持单 worker，在固定源码指纹下运行 100-state catalog、query 与缓存恢复。
2. 每个状态的诊断查询是一个最多 16 个动作的稳定前缀：先纳入 canonical source 的唯一 observed ActionKey 和 `SKIP` 并去重，再对其余 canonical ActionKeys 按 `hash(seed, state_hash, ActionKey)` 的稳定顺序做 uniform 补齐。记 $B_s=\min(16,C_s)$，最终必须恰好查询 $B_s$ 个唯一动作；该前缀不使用 footprint 或 $Q_H$。
3. 只有 100 个状态、1,570 个计划 query 全部成功且结果 identity 验证通过时，执行状态才是 `complete`，此时才允许计算 Signal gate；不删除失败状态，不用新状态补数。
4. 对 signal-eligible 状态，在已查询 $Q_H$ 上使用 Hyndman–Fan Type 7 线性分位数，定义 spread 为 $P_{90}^{(7)}-P_{10}^{(7)}$；当且仅当 spread 不低于 0.01 时记为有非平凡信号。
5. Signal 的工程 Go 条件为：signal-eligible 至少 80/100；其中 spread 不低于 0.01 的总通过率不低于 60%；且 `cities_08` 与 `cities_04` 各自的通过率均不低于 50%。分母分别是全部 signal-eligible 状态和该 instance 内的 signal-eligible 状态。
6. 从 dev 预先选定至少 20 个满足 $17\le C_s\le128$ 的去重状态，对全部 canonical actions 做 exhaustive $Q_H$；状态选择只使用 instance、$C_s$、step 和 hash，不使用已见标签。
7. 从 exhaustive 集合中预先选定 10 个状态，单独比较 objective-greedy 与 beam-8 continuation 的动作排序。MIP continuation 只在可解状态上作为另列的附加参考，分开报告状态数、超时与相关性，不与 beam-8 合并成一个门槛数字。
8. 所有状态重放两次并比较 hashes；报告全部连续 spread、分 instance 通过率、查询失败、wall-time、peak memory 与 cache hit，不只报一个二值结论。

Mac 执行护栏固定为单 worker、每 state 最多 2 次尝试、query timeout 120 秒、state 累计 timeout 1,200 秒、worker peak RSS 上限 6,144 MiB。执行语义严格区分：

- 尚未运行，以及 `query_timeout`、`state_timeout`、`rss_exceeded`、`worker_exit`、`worker_error`、`attempt_exhausted`、`interrupted` 均记为 `incomplete + not_evaluated + method_decision=null`；即使尝试预算耗尽，也只表示标签缺失，不是方法 No-Go；
- `identity_mismatch`、`cache_corrupt`、`hash_mismatch`、`frozen_drift`、`runner_drift` 记为 `invalid + not_evaluated + method_decision=null`；`invalid` 为粘性终态，修复后必须使用新的 run directory 保留原始证据；
- 只有 `complete` 才根据冻结阈值得到 `pass/go` 或 `fail/no_go`。任何缺失或非法 query 都不能静默跳过、替换状态或进入 gate 分母。

这一诊断只有 2 个 dev instance clusters，因此上述条件只是决定是否继续工程投入的 Go/No-Go，不是统计推断。禁止为 100-state 通过率报告独立同分布假设下的 confidence interval、$p$ 值或“显著”声称；状态数不能替代独立 instance 数。

### 7.3 500-state train→dev selector gate

- 500 个拟合状态只能从六个 train instances 产生，并按 `state_hash` 去重；
- footprint scaling、模型拟合、BC 初始化与优化步骤只使用 train；
- `cities_08` 与 `cities_04` 只用于预注册 exhaustive selector 指标、模型选择和完整 schedule gate，不进入梯度更新；
- sealed test 不读取 schedule、trace、标签或模型结果；
- 比较 Uniform-K、Policy-sampling-K、Policy-top-K、generic feature farthest-first 和 CCOD；所有 sampled 方法共享相同 train states、候选集、continuation、模型、loss、查询上限、种子与训练步数；
- selector-only 主指标在预注册的 20 个 exhaustive dev states 上计算 $\operatorname{BTailCov}_{16}$；
- 只有 CCOD 相对最佳通用 selector 的 mean $\operatorname{BTailCov}_{16}$ 至少提高 0.10，且 `cities_08`、`cities_04` 两个 instance-level mean difference 均严格大于 0，同时 train→dev closed-loop 满足门槛，才扩展到 paper set；
- 若只提高离线排序而不改善完整 schedule 趋势，则方法性 No-Go。

### 7.4 30–60-instance paper set 与 2,000-state training set

只有 500-state gate 通过后才建立 3 个规模档、共 30–60 个独立 instances 的 paper set。在产生该集合的任何 $Q_H$ 标签前，冻结 semantic identity、按规模分层的 train/dev/sealed-test 划分、objective、source family、状态抽样规则、feature schema、continuation、$\epsilon_{\mathrm{pair}}$、seeds 与 evaluation manifest。

2,000 个训练状态只能来自 paper-set train fold；paper dev 只用于模型选择，paper test 在最终方法、超参数和 checkpoint 选择规则锁定前保持封存。strong solver 只在预先指定的小规模或代表性子集运行；不得复制同一 instance 的不同 solver trace 来增加独立样本数。

只保留 500-state gate 中最强的两个通用 selectors 与 CCOD；labels 按 `(state_hash,candidate_key)` 独立缓存，$B\in\{4,8\}$ 复用 $B=16$ 稳定前缀。在 Mac 上先完成 64-state overfit、一个 seed 的 CPU pilot 和 learned-scheduler feasibility；通过后才准备 3090 的紧凑训练包。最终 CI 与显著性分析只在该 30–60-instance paper set 上进行，并以 instance 为 cluster。

不默认扩展到全部 7,500 或 20,500 rows。是否继续扩展只由独立 instance 覆盖和学习曲线决定。

### 7.5 最低学习基线

#### Imitation baselines

- BC-Single；
- BC-All；
- group-normalized quality-weighted BC。

#### Cost-sensitive baselines

- SeaRNN-style Uniform-K；
- SeaRNN-style Policy-sampling-K；
- SeaRNN-style Top-K；
- generic feature farthest-first；
- Herrmann-style direct $Q_H$ regression；
- 小型 exhaustive states 上的 full-action LOLS/AggreVaTe-style cost-to-go 上界；
- CCOD footprint selector。

所有 sampled cost-vector 方法共享模型、label oracle、loss、预算和训练步骤，只替换 selector。

#### Scheduling references

- profit-first / completion-first / balanced greedy；
- MIP（只在可解规模）；
- SA / GA / ACO；
- PPO（仅在已有可复现 checkpoint/结果的场景）。

这些方法主要提供 schedule quality–runtime 参照，不要求在 3090 上重训。

### 7.6 主指标

#### 完整 schedule

- Total Profit；
- Task Completion Rate；
- 原始 TM（越小越好）与 $1-\mathrm{TM}$（越大越好）；
- Resource Balance；
- combined $F_\omega$；
- Feasibility Rate；
- online wall-clock 与 peak memory。

主指标是 held-out instance 上的 $F_\omega$ 差值；单步 top-1 accuracy 只作辅助诊断。

#### 查询与标签效率

- $\operatorname{BTailCov}_{16}$（仅 exhaustive dev states 的 selector 主指标）；
- worst-decile regret recall@B；
- best-decile utility recall@B；
- NDCG / Spearman；
- 每 1,000 次 counterfactual queries 的完整 schedule $F_\omega$ 增益；
- Mac CPU label wall-time；
- cache hit、failure 和 duplicate-prefix rate。

#### 统计

- 本节的 cluster bootstrap 与显著性分析只用于后续 30–60 个独立 instances 的 paper set，不用于只含 2 个 dev clusters 的 100-state 工程诊断；
- final shortlisted methods 使用 5 个训练 seeds；
- paired comparison 以同一 held-out instance 和 objective 为配对；
- cluster bootstrap 以 instance 为重采样单位；
- 报告 95% confidence interval 与 effect size；
- feasibility 单独报告，不能用 objective 增益掩盖不可行 schedule。

### 7.7 最小消融

只保留能回答核心问题的消融：

1. $B\in\{4,8,16\}$；
2. $H=5$ 为主，$H=10$ 只在小诊断集检查稳定性；
3. 去掉 conflict footprint；
4. 去掉 resource footprint；
5. footprint-permutation 主负对照；
6. 独立 MLP 与集合条件 scorer；
7. direct $Q_H$ regression 与 pairwise ranking。

删除旧方案中的 $K\in\{32,64\}$、$H=20$、$\beta$ 大网格、双头权重网格和 learner-state 主实验。

### 8.4 何时才允许租 3090

1. P0 全部通过；
2. 100-state 执行状态为 `complete` 且 Signal gate 为 `pass`；
3. 预注册的 20 个 exhaustive states 全部完成，固定 10 个状态的 greedy/beam-8 median Spearman 不低于 0.70；任一相关系数未定义即失败；

7. Mac 上 64-state overfit 的 pairwise accuracy 不低于 0.95；
8. train→dev、至少 500-state、一个 seed 的 CPU pilot 达到 validation pairwise AUC 不低于 0.55、Spearman 不低于 0.10；任一指标未定义即失败；
9. `cities_08` 与 `cities_04` 的 closed-loop $F_\omega$ delta 均不低于 0，且两者平均值严格大于 0；feasibility 为 100%；
10. 上述本地训练 wall-time 不超过 3,600 秒、peak RSS 不超过 6,144 MiB；

正式 100-state 配置中 `pilot_closed_loop_objective_delta_min=0.0` 只是单实例非负机器护栏；租卡授权采用上面更严格的“两实例均非负且均值严格为正”，不会把零收益当作租卡理由。

### 8.5 3090 使用方式

3090 只用于模型训练，不用于：

- 生成 counterfactual labels；
- 跑 MIP / SA / GA / ACO；
- 状态重放；
- 最终 schedule evaluation；
- 临时清洗 26 GiB JSONL。

建议一次短租分三段：

1. **30 分钟 smoke**：一个 seed、一个 epoch；出现 NaN/OOM、GPU 利用率长期低于约 30–40%、数据等待大于计算时间，立即停止；
2. **1–2 小时 pilot**：enriched BC 与 CCOD 各一个 seed；held-out ranking 不为正或 decoder 失败，立即停止；
3. **4–8 小时正式 block**：只跑入选模型、5 seeds 和最小消融；每 epoch checkpoint，连续 2–3 个 epoch 无改进即 early stop。

租用结束前下载 checkpoint、optimizer state、训练日志、配置、环境清单和 checksums。完整 schedule rollout 回 Mac 执行。

由于模型目标小于 0.5M 参数，如果缓存后的 Mac CPU 训练已经足够快，允许完全不租 3090；GPU 是缩短正式多 seed 训练的工具，不是论文成立条件。

## 11. 主要风险与应对

### 风险 1：cheap continuation 排序不可信

应对：小状态 exhaustive + beam-8 rank stability；MIP 仅在可解状态上单列参考。失败时只修 label oracle，不训练更大模型。

### 风险 2：footprint 只是通用 diversity 的换名

应对：加入 generic feature farthest-first、公平 SeaRNN-style selectors 和 selector-only permutation；贡献只由相同查询上限下的 $\operatorname{BTailCov}_{16}$ 与下游增益共同支撑。

### 风险 3：现有 trace 跨实例覆盖不足

应对：100/500-state 只做诊断；方法通过后才建立 30–60 instance 的紧凑多规模 paper set。

### 风险 4：off-trace distribution shift

应对：最终评估必须从空 schedule closed-loop 运行。若 only-trace 学习的收益在早期消失，记录 state drift；只有主方法已经有正信号时，才考虑小规模 learner-state augmentation，不把它预设为主贡献。

### 风险 5：候选 cap 改变最优动作

应对：始终写 $\mathcal A_C$；在小实例上做 cap stability；正式比较中所有方法共享 $C$。

### 风险 6：标签生成成为真正瓶颈

应对：100 → 500 → 2,000 gate、prefix 去重、query union 缓存、$B$ 前缀复用、2–4 workers、每 state checkpoint。禁止直接运行旧方案的 $7{,}500\times64\times20$。

### 风险 7：GPU 实际无加速

应对：先制作 numeric shards；30 分钟 smoke 有明确 kill switch；若 GPU 等 I/O 或相对 Mac 无实质加速，立即释放。

1. **问题与数据协议**：将 heterogeneous solver final schedules 明确定义为 canonical solution-induced constructive traces，并建立精确重放、instance-level split 和 capped dynamic action set 上的 partial cost-vector supervision 协议；
2. **方法**：提出 constraint-consistent 的 EOS conflict/resource footprint query allocation，在每状态至多 $B$ 次反事实查询下提高 $Q_H$ 的容量平衡双尾覆盖与排序监督效率；
3. **实证**：在 EOS-Bench 上联合评估完整 schedule quality、online runtime 与 Mac CPU offline labeling cost，并通过 exhaustive small states、strong-continuation rank check 和 selector-only permutation 证伪协议验证机制。

双头网络、regret penalty decoder、learner-state aggregation 和“首次反事实蒸馏”均不写入贡献。
