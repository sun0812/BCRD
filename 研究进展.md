## 进展 005：确定性 CCOD Continuation Oracle

日期：2026-07-15
状态：已完成

### 对应提交

- `c63929a`：`feat: add deterministic CCOD continuation oracle`
- `717fdc2`：`feat: harden and batch CCOD continuation queries`

### 完成内容

1. 按约定在独立目录 `algorithms/ccod/` 新增算法，实现规范 capped 候选集上的 `force_action(s,a)`、显式 SKIP 与 objective-greedy 有限视野 continuation。
2. 强制动作严格计作第一个决策；默认 $H=5$，若剩余任务不足则只执行到任务耗尽。标签严格定义为：

$$
Q_H(s,a)=F_\omega\!\left(\mathcal C_H(s\oplus a)\right)-F_\omega(S(s)),
$$

不额外叠加 SKIP penalty。
3. 平局只使用 canonical ActionKey 的字典序打破，输入 schedule、候选与嵌套 `sat_angles` 均使用防御性副本。
4. 删除可由调用方注入 `allowed_actions` 的公开绕过入口；公开 `force_action` 必须重新枚举当前 canonical capped feasible set，cap 外动作直接拒绝。
5. 查询前重新核对 schedule、objective、candidate order/membership、physical/state hash、constraint/enumerator/objective 配置身份与运行时对象指纹，避免“旧 state hash + 已变化 schedule”污染标签缓存。
6. label cache 使用独立 query key，身份同时覆盖状态、ActionKey、$H$、三类配置哈希与关键实现文件集合哈希；写入前重算最终调度指纹，篡改结果拒绝入库。
7. 将 greedy 路径从逐候选重复扫描完整候选集的 $O(C^2)$ 结构改为一次规范枚举后的 $O(C)$ 候选遍历；目标计算与约束复核成本仍保留，等待真实 profile 决定是否做增量优化。
8. 目标权重现在 fail-fast 拒绝负数、非有限值和全零配置，与方案中的 $\omega_i\ge 0$ 一致。

### 验收证据

- 固定环境：Python 3.10.20。
- 全仓回归：61/61 通过；其中 CCOD 共 32 项测试。
- 已覆盖：$H\in\{1,5\}$、强制动作计步、SKIP、任务末尾提前终止、observed prefix 对齐、cap 外动作拒绝、三类配置失配、候选顺序/成员变化、嵌套姿态角篡改、query key 配置敏感、缓存往返与最终调度篡改拒绝。
- 用户原有、已暂存的 `方案.md` 未进入本阶段提交。

## 进展 007：100-state 诊断预注册冻结

日期：2026-07-15
状态：已完成；预注册规则、自动化生成器与完整性守卫已提交

### 冻结内容

1. 明确独立单位是语义 scenario/instance，不是 JSONL row 或 solver schedule。冻结划分为 `train=[03,05,01,09,06,07]`、`dev=[08,04]`、`sealed_test=[02,10]`；同一语义 instance 的全部 solver、objective、source file 和 trace row 必须位于同一 fold。
2. instance manifest 以规范化 scenario 内容的 SHA-256 为真实 identity，同时记录 alias、raw file hash、source stem 和内部 scenario id。alias 不参与判定是否为同一实例；sealed test 在方法锁定前不读取 schedule 或标签。
3. 100-state 诊断只使用 balanced objective 下的 SA/GA/ACO solution-induced traces，并且只使用 `cities_08` 与 `cities_04`。按 `state_hash` 去重后每个 dev instance 抽 50 个、共 100 个 actionable states；要求 `step <= 95` 且 candidate count 至少 2。
4. 将 candidate count 至少 10 的状态定义为 signal-eligible，预注册目录中至少有 80/100 个此类状态。如果 catalog 在标签前无法满足该条件，则报告覆盖不足并停止，不在看到 $Q_H$ 后换样本。
5. 每状态先按 `hash(seed, state_hash, source_family)` 冻结一个不依赖 observed 内容的 canonical source，再以该来源的唯一 observed ActionKey 和 `SKIP` 为锚点；去重后按 `hash(seed, state_hash, ActionKey)` 的稳定 uniform 顺序补齐到最多 $B=16$。候选少于 16 时查询全部候选。这一首轮前缀不使用 footprint 或已见标签。
6. 只有 100 个状态、1,570 个计划 query 全部成功且 identity 校验通过，执行状态才是 `complete` 并允许判定 Signal gate；后续执行器将资源/进程性缺标签细分为 `incomplete`，将身份/哈希/漂移破坏细分为 `invalid`，两者都不删除失败状态、不补数、不产生方法结论。
7. 对 signal-eligible 状态使用 Hyndman–Fan Type 7 线性分位数；非平凡信号定义为 $P_{90}^{(7)}-P_{10}^{(7)}\ge0.01$。工程 Go 条件为总通过率至少 60%，且两个 dev instances 各自至少 50%。
8. 强标签核验限定为至少 20 个 $17\le C_s\le128$ 的 exhaustive states，其中预选 10 个做 beam-8 排序对比。MIP 只在可解状态上单列报告，不与 beam-8 合并成同一门槛。
9. Sats20 只作 candidate-scale、内存和 wall-time 压力测试，不进入 100-state 目录或 Go/No-Go 分母。由于当前只有 2 个 dev clusters，这一诊断只支持工程 Go/No-Go，禁止 confidence interval、$p$ 值与显著性声称。
