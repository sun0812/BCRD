## 进展 005：确定性 CCOD Continuation Oracle

日期：2026-07-15
状态：已完成

### 对应提交

- `c63929a`：`feat: add deterministic CCOD continuation oracle`
- `717fdc2`：`feat: harden and batch CCOD continuation queries`

### 完成内容

1. 按约定在独立目录 `algorithms/ccod/` 新增算法，实现规范 capped 候选集上的 `force_action(s,a)`、显式 SKIP 与 objective-greedy 有限视野 continuation。
2. 强制动作严格计作第一个决策；默认 $H=5$，若剩余任务不足则只执行到任务耗尽。标签严格定义为：

$$
Q_H(s,a)=F_\omega(S_H(s,a))-F_\omega(S(s)),
$$

不额外叠加 SKIP penalty。
3. 平局只使用 canonical ActionKey 的字典序打破，输入 schedule、候选与嵌套 `sat_angles` 均使用防御性副本。
4. 删除可由调用方注入 `allowed_actions` 的公开绕过入口；公开 `force_action` 必须重新枚举当前 canonical capped feasible set，cap 外动作直接拒绝。
5. 查询前重新核对 schedule、objective、candidate order/membership、physical/state hash、constraint/enumerator/objective 配置身份与运行时对象指纹，避免“旧 state hash + 已变化 schedule”污染标签缓存。
6. label cache 使用独立 query key，身份同时覆盖状态、ActionKey、$H$、三类配置哈希与关键实现文件集合哈希；写入前重算最终调度指纹，篡改结果拒绝入库。
7. 将 greedy 路径从逐候选重复扫描完整候选集的 $O(C^2)$ 结构改为一次规范枚举后的 $O(C)$ 候选遍历；目标计算与约束复核成本仍保留，等待真实 profile 决定是否做增量优化。
8. 目标权重现在 fail-fast 拒绝负数、非有限值和全零配置，与方案中的 $\omega_i\ge 0$ 一致。

### 验收证据

- 固定环境：Python 3.10.20。
- 全仓回归：61/61 通过；其中 CCOD 共 32 项测试。
- 已覆盖：$H=1/H=5$、强制动作计步、SKIP、任务末尾提前终止、observed prefix 对齐、cap 外动作拒绝、三类配置失配、候选顺序/成员变化、嵌套姿态角篡改、query key 配置敏感、缓存往返与最终调度篡改拒绝。
- 用户原有、已暂存的 `方案.md` 未进入本阶段提交。

### 当前边界

## 进展 006：真实 Sats1/M100 的 10-state CCOD 冒烟

日期：2026-07-15
状态：已完成

### 对应提交

- `717fdc2`：`feat: harden and batch CCOD continuation queries`

### 固定输入

- 场景：`Scenario_Sats1_M100_T7.0d_cities_01.json`；
- 轨迹：`c2_profit_first_implicit`；
- 状态步：`0,11,22,33,44,55,66,77,88,99`；
- 动作：observed 与 SKIP，去重后 17 个唯一 query；
- continuation：$H=5$，canonical cap $C=8192$；
- 两个全新进程：不同 `PYTHONHASHSEED`；
- 固定实现指纹：`sha256:30ae07f824148447c1f1ea7e9385f89c96794afa6d5ddc369b6564db75dfb5f4`。

### 验收结果

1. 17/17 个 query 的 query、result、rollout 与 final schedule hashes 在两个进程中完全一致，一致率为 100%。
2. force、query、cache 失败数均为 0；第二个进程 17/17 命中第一个进程写入的内容寻址缓存。
3. step 0–88 的 9 个 observed 动作全部精确复现下一 prefix；step 99 没有下一状态，正确只执行 1 个 SKIP 决策并以 task exhaustion 结束。
4. 抽查状态候选数为 `391,710,1,332,1450,887,1,1538,494,1`，cap hit 为 0。
5. 单 query wall-time：median 0.188 s、P90 0.536 s、max 0.545 s。
6. 每状态一次性 `prepare`：median 0.103 s、P90 0.371 s；exact prefix restore：median 1.762 s、P90 4.553 s、max 5.214 s。
7. 两个 worker 总 wall-time 分别为 28.09 s 与 28.56 s；峰值 RSS 分别为 242.1 MiB 与 257.2 MiB，远低于 6 GiB 护栏。
8. 使用论文主目标 `c3_sa_p0.25_c0.25_t0.25_b0.25` 复核同一组状态，17/17 个 query 仍跨进程完全一致；query median/P90/max 分别为 0.201/0.526/0.545 s，prepare median/P90 为 0.101/0.355 s，峰值 RSS 不超过 252.0 MiB。

### 过程异常与处理

第一次完整运行被一致性检查拒绝，因为两个 worker 之间关键实现文件从旧指纹 `d44f…` 更新到 `30ae…`。这不是同一源码下的随机漂移，也未被当作通过结果。smoke 现已增加“worker 实现指纹必须等于启动指纹、运行期间源码不得变化”的 fail-fast 检查；在稳定源码上重新运行后得到上述通过结果。

### 当前判断

P0 correctness 的真实小样本门槛已经通过，且单 worker 内存余量充足。下一步仍使用 1 worker；先冻结 instance split 并做 100-state signal diagnostic，不因为本次耗时较低就提前租 3090。
